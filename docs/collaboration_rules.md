# 团队协作规则

## 1. 文档目的
本文件用于说明项目的基础协作规则。  
目标是减少混乱，提高效率，并保证团队成员在 Demo 开发阶段能够更清晰地协作。

当前项目的首要目标，是在大约一个半月内完成一个**可游玩、可展示的 Demo**。

---

## 2. 核心原则
现阶段所有决策都应优先服务于 Demo 目标。

这意味着：
- 优先保证完成，而不是一开始追求完美
- 优先做出可运行、可展示的结果，而不是堆过多理想化内容
- 对开发周期过长的内容进行简化
- 避免选择当前阶段风险过高、实现难度过大的技术方案

团队应先建立一个稳定、可展示的核心体验，再考虑后续扩展。

---

## 3. 文件与资源管理
从开发开始阶段起，团队应保持清晰的文件结构与资源管理方式。

### 3.1 文件夹结构
团队成员应遵守统一的项目文件夹结构，不应随意放置文件。

主要文件夹包括：
- `game/`：Godot 项目本体与游戏资源
- `docs/`：正式项目文档
- `meetings/`：会议记录
- `planning/`：开发计划、里程碑、任务拆分
- `references/`：灵感图与参考资料
- `marketing/`：宣传或对外展示相关内容

### 3.2 命名规则
文件命名需要清晰、统一、有实际含义。

规则建议：
- 尽量使用英文命名
- 避免使用含义不清的文件名，如 `new file`、`最终版最终版`、`test2`、`最新版`
- 使用有意义的命名，例如：
  - `story_outline_v1.md`
  - `bg_city_night.png`
  - `player_movement.gd`

### 3.3 正式文件与临时文件分开
临时测试文件与正式使用文件应尽量区分存放。

例如：
- 临时测试资源不要和正式资源混在一起
- 未完成草稿要有明确标记
- 过期文件应删除或归档，而不是长期堆放在项目中

---

## 4. GitHub 与版本管理
本项目使用 GitHub 进行版本控制与团队协作。

### 4.1 基本流程
基础工作流程如下：
1. 在本地进行修改
2. 使用清晰的提交说明进行 commit
3. 将修改 push 到 GitHub

### 4.2 提交说明
每次提交都应写清楚本次改动内容。

好的例子：
- `Add initial dialogue system`
- `Update opening scene background`
- `Fix player movement bug`

不好的例子：
- `update`
- `aaa`
- `fix`
- `123`

### 4.3 修改重要文件前先沟通
如果成员需要修改重要的公共文件，应提前和团队沟通，尤其是以下内容：
- 核心场景
- 公共脚本
- 重要美术资源
- 故事主线文档

### 4.4 同步工作
团队成员在开始重要开发前，应先同步最新版本，减少文件冲突风险。

---

## 5. 会议记录
每次会议都应保留文字记录。

会议记录至少应包含：
- 日期
- 参与成员
- 讨论主题
- 已确定的结论
- 分配的任务
- 下一次检查时间或截止时间

建议文件名格式：
- `2026-04-03-kickoff.md`
- `2026-04-07-demo-scope.md`

会议记录统一放在 `meetings/` 文件夹中。

---

## 6. 任务分工
所有任务应尽量明确分配。

每项主要任务最好包含：
- 要做什么
- 由谁负责
- 预计完成时间
- 当前状态（如有需要）

团队成员不应默认“别人应该会做”，除非任务已经被明确分配。

---

## 7. 范围控制
项目开发过程中应避免范围失控。

如果某项内容：
- 开发时间过长
- 技术难度过高
- 会带来大量额外工作
- 对当前 Demo 不是必要部分

那么应考虑：
- 简化
- 延后
- 删除

这个原则很重要，因为当前阶段**按时完成 Demo**比塞进所有好想法更重要。

---

## 8. 开发时间记录
建议每位成员保留简单的个人开发记录。

记录内容可以包括：
- 日期
- 花费时间
- 完成的任务
- 简短进展说明

这样做有助于：
- 项目管理
- 进度复盘
- 后续贡献评估

---

## 9. 贡献与成果分配
如果项目未来产生实际价值、收入或其他回报，原则上应按贡献进行分配。

贡献可综合参考：
- 实际完成工作量
- 工作质量
- 投入时间
- 承担责任的程度
- 对整体推进的重要性
- 解决关键问题的贡献

团队应尽量在开发过程中保留贡献记录，而不是等项目结束后再临时讨论。

---

## 10. 沟通原则
团队沟通应保持直接、尊重、务实。

出现问题时应：
- 尽早提出
- 避免默认别人已经知道
- 尽量清楚讨论解决方案
- 对重要决定进行记录

目标不仅是完成任务，也是保证协作本身可以持续进行。

---

## 11. 当前优先级
在当前阶段，项目优先级顺序如下：

1. 做出可游玩的 Demo
2. 保证核心体验清晰
3. 保持项目结构整洁
4. 控制开发范围，减少不必要复杂度
5. 若时间允许，再继续打磨细节

---

## 12. 最后说明
本文件为当前项目阶段的协作规则草案。  
后续如果团队规模扩大、项目范围变化，或项目进入下一阶段制作，可再进行更新。
