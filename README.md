# me-and-me-gamedemo
GameDemo project for 《我和我》
《我和我》Demo阶段团队协作原则（初稿）
1. 资源管理与文件规范

文件格式、命名方式和资源存放结构非常重要。
在开发前期建立清晰统一的资源管理规范，可以显著减少后续开发中的混乱，也能提高开发者之间交接、协作和查找文件的效率。

建议包括：

统一文件命名规则
明确不同资源的文件夹分类
区分正式版、测试版、废弃版文件
重要资源尽量避免重复存放和随意改名
2. GitHub协作与版本同步

团队开发过程中，建议每位开发者都使用 GitHub 进行项目同步与版本管理。
这样可以确保：

每个人的改动有记录
不同开发者之间方便同步最新版本
出现问题时可以回退到之前版本
降低文件互传导致的混乱和覆盖风险

如果后续团队扩大，GitHub 也能作为协作基础工具继续使用。

3. 开发过程记录制度

为了保证项目推进清晰可追踪，团队需要建立基本的过程记录机制。

3.1 会议记录

每次讨论会议都应保留记录。
记录内容包括：

讨论了什么
确定了什么
哪些问题暂时未解决
谁负责后续执行

这样可以避免“说过但没人记得”的情况，也方便后面复盘。

3.2 个人开发时间记录

每位开发者需要记录自己的开发时间和主要工作内容。
这不仅方便项目管理，也能作为后续贡献评估的重要依据。

3.3 任务分工与开发计划

团队需要提前确定整体开发计划，并明确：

每个章节或模块要做什么
每位开发者负责哪些具体任务
每个阶段的目标和截止时间

所有任务最好尽量具体，避免出现“大家都以为别人会做”的情况。

4. Demo阶段的核心开发原则

本项目现阶段的首要目标，是在一个半月内完成可展示的 Demo。
因此，开发中的所有选择都应优先服务于这个目标。

原则上：

以“能做出来、能跑起来、能展示”为第一优先级
某些支线内容如果开发周期过长，应考虑删除或简化
某些技术方案即使效果很好，但实现难度过高、风险太大，也应及时舍弃
Demo阶段追求的是“完成度”和“可落地性”，而不是一次性做到最完整

简单说就是：
先做出能成立的核心体验，再考虑扩展。

5. 成果与回报分配原则

如果项目最终有实际产出，并产生收益或回报，原则上按照成员的实际贡献进行分配。

贡献评估可综合参考：

实际完成的任务量
工作质量与完成度
投入时间
关键问题解决情况
对整体推进的实际作用

为了避免后续争议，建议团队在前期就尽量把“贡献记录”做清楚，而不是等到项目结束后再回头判断。

文档怎么命名最清楚

我建议你统一一个命名规则：

规则1：全英文或拼音，不要中英乱混

比如不要：

第一章最终版new.docx
剧情最新版2真的最新版.md

建议这样：

chapter1_outline.md
story_outline_v1.md
demo_scope_v2.md
规则2：日期型文档就用日期开头

适合会议记录、周报、决策记录：

2026-04-03-kickoff.md
2026-04-08-weekly-update.md
规则3：版本型文档用 v1 v2 v3

适合会改来改去的策划文档：

story_outline_v1.md
story_outline_v2.md

不要写：

最终版
真最终版
最终版修改
最终版2
